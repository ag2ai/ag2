A2UI **actions** are the clickable parts of a UI — buttons that, when pressed, drive the next turn. A clickable button is an `#!python @a2ui_action` tool you **declare on the transport** via `#!python actions=[...]` — the agent stays a plain `#!python Agent`. The transport injects the tool for the turn, advertises the button to the LLM, and routes a click back to it.

## Tool-Backed Buttons (`@a2ui_action`)

Decorate a tool function with `#!python @a2ui_action`, then declare it on the server with `#!python actions=`. It becomes both a tool the agent can call **and** a server `#!python event` action whose name routes a click back to that tool:

```python linenums="1" hl_lines="6-9 13"
from autogen.beta import Agent
from autogen.beta.a2ui import A2UIServer, a2ui_action
from autogen.beta.a2ui.transports import RestTransport
from autogen.beta.config import AnthropicConfig

@a2ui_action(description="Schedule all posts for the given time")
def schedule_posts(time: str) -> str:
    # ... do the work ...
    return f"scheduled all posts for {time}"

agent = Agent(name="ui_agent", config=AnthropicConfig(model="claude-sonnet-4-6"))

app = A2UIServer(agent, transport=RestTransport(encoding="jsonl"), actions=[schedule_posts])
```

When the LLM renders a button whose `#!python action.event.name` is `#!python "schedule_posts"`, a click sends the button's `#!python event.context` back as the tool's arguments. The decorator derives an example context from the function's parameter schema (here `#!python {"time": "<string>"}`) and injects it into the system prompt so the LLM emits the right keys; pass `#!python example_context=` to override it.

Declaring the button on the transport — not on the agent — keeps the `#!python Agent` plain and reusable across deployments. The same `#!python actions=[...]` is accepted by `#!python A2UIAgentExecutor` for the [A2A transport](/docs/beta/a2ui/a2a), so a button is declared the same way wherever the agent is served.

## Other Kinds of Buttons

A2UI components support two button shapes the spec distinguishes by what the click does:

- **Server `#!python event`** — sends a client→server message; the server handles it. This is what `#!python @a2ui_action` produces.
- **Client `#!python functionCall`** — runs a function **on the renderer with no network round-trip** (e.g. `#!python openUrl`, copy-to-clipboard).

Client `#!python functionCall` buttons execute entirely in the client, so they are **not registered on the server**. The LLM learns which client functions it may target from the active **catalog** and from the client's advertised [capabilities](/docs/beta/a2ui/capabilities) — not from server-side code. You can also render a button whose `#!python event.name` isn't backed by a tool: a click on it is rewritten generically into the next prompt so the LLM simply continues the conversation.

## The Click Round-Trip

The server is stateless, so a click is just another request. The client posts the conversation plus an `#!python a2ui` envelope describing what was pressed; the transport rewrites it into the current turn:

```python linenums="1" hl_lines="4-15"
import httpx

body = {
    "messages": [],  # plus any prior conversation
    "a2ui": [
        {
            "version": "v0.9",
            "action": {
                "name": "schedule_posts",
                "surfaceId": "s1",
                "sourceComponentId": "schedule_btn",
                "timestamp": "2026-06-17T12:00:00Z",
                "context": {"time": "2:00 PM"},
            },
        }
    ],
}

async with httpx.AsyncClient(base_url="http://127.0.0.1:8000") as client:
    resp = await client.post("/a2ui", json=body)
```

When the action name matches a registered `#!python @a2ui_action` tool, the click resolves to that tool and runs it; otherwise the click is rewritten into a plain instruction so the LLM continues the conversation.

!!! note "v1.0 round-trip"
    With `#!python protocol_version="v1.0"`, a client's reply to a server `#!python callFunction` arrives as a `#!python functionResponse` envelope in the next request and is rewritten to a continuation prompt — the stateless transport needs no pause/resume bookkeeping. The same applies to `#!python actionResponse`.

## Observing Client Interactions

Each client→server interaction the server receives — a button `#!python action`, a v1.0 `#!python functionResponse`, or an `#!python error` — is surfaced on the turn's stream as an `#!python A2UIClientEvent`, the inbound mirror of `#!python A2UIMessageEvent`. Subscribe to react to or log client interactions (beyond what the LLM sees in its prompt):

```python linenums="1"
from autogen.beta.a2ui import A2UIClientEvent

@stream.subscribe
async def on_client_event(event):
    if isinstance(event, A2UIClientEvent):
        log.info("client interaction: %r", event.interaction)
```

!!! note
    A purely client-side `#!python functionCall` button (e.g. `#!python openUrl`) runs on the renderer with no network round-trip per the A2UI spec, so it produces no envelope and therefore no `#!python A2UIClientEvent`.

## Observing Validation Failures

A2UI has no server→client error message, so when the model can't produce schema-valid UI within `#!python validation_retries + 1` attempts, the agent **gracefully degrades** to prose and emits no `#!python A2UIMessageEvent`. To tell "couldn't build UI" apart from "intentionally answered with text", subscribe to `#!python A2UIValidationFailedEvent` on the stream:

```python linenums="1"
from autogen.beta.a2ui import A2UIValidationFailedEvent

@stream.subscribe
async def on_validation_failed(event):
    if isinstance(event, A2UIValidationFailedEvent):
        log.warning("A2UI validation failed after %d attempts: %s", event.attempts, event.errors)
```
