# AG2 Network V3: Session-Based Architecture

## Status

**IMPLEMENTED.** All phases complete. Legacy network code removed. 28 source files, 289 tests.

Supersedes the network layer described in `ag2_network_framework.md`. Framework core (Agent, Actor, Stream, Watch, Observer, Assembly, Knowledge, Compact, Aggregate) is unchanged — this redesign targets only `autogen/beta/network/`.

---

## Problems With Current Architecture

1. **No stateful communication.** Every `hub.delegate()` is fire-and-forget. Actors cannot have multi-turn conversations. No way to maintain context across exchanges.

2. **Scattered interfaces.** Network tools, delegation, topics, background tasks, knowledge queries — each has its own API shape. No coherent interaction model.

3. **Topology/Plugin complexity.** Pipeline, Fanout, Conditional, RouteDecision, Plugin protocol with install/uninstall/process lifecycle — powerful but high cognitive barrier. Developers must understand routing composition to do basic access control.

4. **No durability or async task tracking.** Background delegations use ephemeral StateStore entries. No replay, no checkpoints, no structured lifecycle. No streaming support for cross-actor responses.

5. **No access control.** Any actor can delegate to any peer. No rules governing who talks to whom, what session types are allowed, or what rate limits apply.

---

## Architecture Overview

Seven concepts replace the current fourteen:

| Concept | What | Replaces |
|---------|------|----------|
| **Hub** | Network center with filesystem-backed state | Hub (simplified), Registry, StateStore (hub-level) |
| **Identity** | Who an actor is (passport + resume + SKILL.md) | `ActorInfo`, capabilities list |
| **Rule** | What an actor can do (access + limits + transforms) | Topology, Plugin (routing), PriorityScheme, ConflictResolver |
| **Session** | Stateful communication container (6 types) | Delegation, Topics |
| **Envelope** | Message unit within a session | Envelope (simplified) |
| **Task** | Lifecycle-managed work unit within a session | Background delegation state |
| **SessionLog** | Append-only WAL for session events | DelegationRequest/Result events on hub stream |

**Removed:** Topology (Pipeline/Fanout/Conditional), Plugin protocol, RouteDecision, PriorityScheme, ConflictResolver, PriorityChannel, TopicPlugin, TopicInboxPolicy.

**Absorbed:** Topics → broadcast sessions + public subscription. Rate limiting → rules. Telemetry → hub stream subscriptions. Priority routing → session-level config + rules.

## Hub Filesystem

The Hub is a virtual filesystem-backed service. All state lives in a navigable tree that maps to any storage backend (memory, disk, S3, database).

```
{hub_id}/
├── meta.json                                   # Hub config, version, created_at
│
├── actors/
│   └── {actor_id}/
│       ├── identity/
│       │   ├── passport.json                   # Required: who this actor is
│       │   ├── resume.md                       # Required: what this actor can do
│       │   └── SKILL.md                        # Optional: how to use this actor
│       └── rule.json                           # Access control, limits, transforms
│
├── sessions/
│   └── {session_id}/
│       ├── meta.json                           # Type, participants, state, config
│       ├── subscriptions.json                  # Non-participant observers
│       └── logs/
│           ├── {date}.jsonl                    # WAL entries, date-partitioned
│           └── tasks/
│               └── {task_id}.jsonl             # Task-specific index
│
└── admin/
    ├── config.json                             # Hub-level defaults, global policies
    └── audit.jsonl                             # Hub-level audit log
```

The filesystem uses `KnowledgeStore` for static data (identity, rules, config) and `SessionLog` for append-only data (session WAL, audit log). See [Persistence Architecture](#persistence-architecture).

## UUID7

UUID7 (RFC 9562) throughout. 48-bit millisecond timestamp prefix gives natural chronological sorting without sequence counters.

| Entity | ID | Human-Readable Name |
|--------|----|---------------------|
| Hub | UUID7 | — |
| Actor | UUID7 (generated by hub at registration) | `owner:name:version` (unique in network) |
| Session | UUID7 | — |
| Envelope | UUID7 | — |
| Task | UUID7 | — |

Actor **name** (`owner:name:version`) is the human-readable identifier. Actor **ID** (UUID7) is the stable reference. The hub maintains the name↔ID mapping. An actor can be re-registered under a new name without breaking existing session references.

Cross-hub references use `hub_id/actor_id` to ensure global uniqueness when federation is added. The `hub_id/admin/` directory is reserved for cross-hub metadata.

Implementation: a `UUID7` type (thin wrapper over `uuid.UUID`) with `.timestamp` property.

## Identity

Every actor registered with a hub has an identity consisting of three parts:

### Passport (required)

Metadata about who the actor is. Machine-readable. Used for registration validation, discovery filtering, and operational constraints.

```python
@dataclass
class Passport:
    actor_id: UUID7                  # Assigned by hub at registration
    name: str                        # owner:name:version (unique in network)
    framework: str                   # "ag2", "langchain", "custom", etc.
    status: Literal["active", "inactive", "suspended"]
    tags: list[str]                  # Searchable tags for discovery
    endpoints: list[str]             # For remote actors (HTTP URLs)
    constraints: PassportConstraints # Max sessions, supported types, pricing, etc.
    created_at: datetime
```

`PassportConstraints` captures operational limits that are intrinsic to the actor (not policy — that's in Rule):

```python
@dataclass
class PassportConstraints:
    supported_session_types: list[SessionType]   # What this actor can handle
    max_concurrent_sessions: int = 100           # Actor's own capacity
    max_message_size_bytes: int = 1_048_576      # Actor's payload limit
    pricing: dict[str, Any] | None = None        # Cost per interaction (informational)
```

### Resume (required, capabilities field is required)

Rich description of what the actor can do. Used by other actors (and their LLMs) to make informed routing decisions. Replaces the flat `capabilities: list[str]`.

```python
@dataclass
class Resume:
    capabilities: list[str]          # Required: searchable keywords
    description: str = ""            # What this actor does
    experience: str = ""             # Past tasks, track record
    knowledge_domains: list[str] = field(default_factory=list)
    tools: list[ToolEntry] = field(default_factory=list)
```

`ToolEntry` describes a tool the actor has:

```python
@dataclass
class ToolEntry:
    name: str
    description: str = ""
    public: bool = True              # Public = discoverable. Exclusive = only this actor has it.
```

The resume is markdown-friendly — it can be rendered as-is in an LLM prompt when another actor inspects this actor's capabilities.

### SKILL.md (optional)

How to use this actor. Written for consumption by other actors' LLMs. Contains best practices for prompting, supported message formats, example interactions. Convention, not schema — free-form markdown.

### Identity dataclass

```python
@dataclass
class Identity:
    passport: Passport
    resume: Resume
    skill: str | None = None         # SKILL.md content
```

## Rule

Rules govern all interactions with an actor. Three concerns: **access**, **limits**, **transforms**.

### Access

Who can interact with this actor and how.

```python
@dataclass
class AccessRule:
    can_initiate_to: list[str] = field(default_factory=lambda: ["*"])
        # Who this actor can start sessions with. Accepts actor names, glob patterns, tags.
    can_receive_from: list[str] = field(default_factory=lambda: ["*"])
        # Who can start sessions targeting this actor.
    can_create_session_types: list[SessionType] = field(default_factory=lambda: list(SessionType))
        # What session types this actor can create.
    can_join_session_types: list[SessionType] = field(default_factory=lambda: list(SessionType))
        # What session types this actor can be invited to or join.
    can_subscribe: bool = True
        # Whether this actor can subscribe to public sessions.
    blocked: list[str] = field(default_factory=list)
        # Explicit deny list (overrides wildcards).
    expose_identity: list[str] = field(default_factory=lambda: ["passport", "resume"])
        # What parts of identity are visible to peers via discovery.
    expose_knowledge_paths: list[str] = field(default_factory=list)
        # What knowledge store paths can be queried by peers (glob patterns).
    allow_sub_sessions: bool = True
        # Whether this actor can create sub-sessions within an existing session.
```

### Limits

Resource constraints enforced by the hub.

```python
@dataclass
class LimitRule:
    max_concurrent_sessions: int = 100
    max_sessions_per_peer: int = 10
    max_message_size_bytes: int = 1_048_576       # 1MB
    rate_limit_per_minute: int = 120
    default_task_ttl_seconds: int = 3600          # 1 hour
    max_task_ttl_seconds: int = 86400             # 24 hours
    max_concurrent_tasks: int = 20
    session_idle_timeout_seconds: int = 1800      # 30 minutes
    response_deadline_seconds: int = 120          # Per session type
    retention_days: int = 30                      # Session log retention
    max_sub_session_depth: int = 10               # Max depth of sub-session chain
```

### Transforms

Pre/post processing of envelopes. Replaces routing plugins (RateLimiter, AuthPlugin, etc.) with per-actor declarative rules backed by registered Python functions.

```python
@dataclass
class TransformEntry:
    name: str                        # Registered transform function name
    condition: dict[str, Any]        # When to apply

@dataclass
class TransformRule:
    inbound: list[TransformEntry] = field(default_factory=list)
    outbound: list[TransformEntry] = field(default_factory=list)
```

Conditions support: `sender` (name/pattern), `sender_tag`, `session_type`, `has_metadata` (key presence), and `custom` (registered predicate name).

Transform functions are registered imperatively on the hub:

```python
async def sanitize_pii(envelope: Envelope) -> Envelope:
    """Strip PII from inbound messages."""
    ...

await hub.register(
    actor,
    identity=identity,
    rule=Rule(transforms=TransformRule(
        inbound=[TransformEntry("sanitize_pii", condition={"sender_tag": "external"})],
    )),
    transforms={"sanitize_pii": sanitize_pii},
)
```

**Distributed execution:** Transforms execute at the boundary of each hub. Outbound transforms run on the sender's hub before the envelope leaves. Inbound transforms run on the recipient's hub after the envelope arrives. Cross-hub transport carries the envelope between outbound and inbound transforms. Each hub only needs transform functions for its local actors — no code shipping between servers.

```
Actor A (Hub 1) sends to Actor B (Hub 2):

Hub 1:  A's outbound transforms → [envelope in transit] → HTTP →
Hub 2:  → [envelope arrives] → B's inbound transforms → deliver to B
```

For transforms that must be understood by both sides (encryption, compression), define **protocol transforms** — built-in functions referenced by name in rules. Both hubs know how to execute them. Custom Python transforms are hub-local only.

### Hub-Level Defaults

The hub has its own config with defaults that actor rules can override within allowed ranges:

```python
@dataclass
class HubConfig:
    hub_id: UUID7 = field(default_factory=uuid7)
    default_session_idle_timeout: int = 1800
    max_session_idle_timeout: int = 7200
    default_response_deadline: int = 120
    global_rate_limit_per_minute: int = 1000
    require_identity: bool = True
    allow_anonymous_sessions: bool = False
    max_actors: int = 10000
    max_sessions: int = 100000
```

### Combined Rule dataclass

```python
@dataclass
class Rule:
    access: AccessRule = field(default_factory=AccessRule)
    limits: LimitRule = field(default_factory=LimitRule)
    transforms: TransformRule = field(default_factory=TransformRule)
```

## Session

A session is a runtime container for stateful, durable, multi-turn communication between participants. Sessions are the primary interaction primitive — all inter-actor communication flows through sessions.

### Key Principle: Network ≠ Harness

**Network sessions are NOT agent sessions.** A network session provides stateful communication and durable logging. It does NOT decide what an actor sees in each LLM round. That is the assembly policy's responsibility. This separation allows flexible context management — an actor might participate in 10 concurrent sessions but only include 2 in its current LLM context window.

The boundary is strict:

| Concern | Responsibility | Layer |
|---------|---------------|-------|
| Store session events (WAL) | Network | `SessionLog` |
| Track session state, participants, tasks | Network | `Hub`, `Session` |
| Invoke actor when a message arrives (push) | Network | `Hub` |
| Expose session data for query | Network | `Hub` API, `session`/`task` tools |
| Decide which sessions appear in LLM prompt | Harness | `AssemblyPolicy` |
| Decide how many messages per session | Harness | `AssemblyPolicy` |
| Decide format (verbatim, summary, structured) | Harness | `AssemblyPolicy` |
| Prioritize sessions when context budget is tight | Harness | `AssemblyPolicy` |

`SessionPolicy` and `TaskStatusPolicy` are **default implementations** — not the only way to consume session data. Developers building complex agents can write their own `AssemblyPolicy` that calls the same hub APIs (`hub.list_sessions()`, `hub.session_log().read()`, `hub.get_task()`) with whatever logic they need. The network doesn't care — it stores and serves data. No assembly logic lives in the network layer.

### Session Types

Six session types, each with a distinct interaction pattern:

| Type | Cardinality | Direction | Pattern | Use Case |
|------|-------------|-----------|---------|----------|
| **notification** | 1→1 | one-way | fire-and-forget with ack | Alerts, status updates |
| **broadcast** | 1→N | one-way | fire-and-forget with per-recipient ack | Announcements, pub/sub replacement |
| **consulting** | 1→1 | structured | one request → one response | Delegation (replaces `hub.delegate()`) |
| **conversation** | 1→1 | bidirectional | free-form, either side sends | Collaboration, ongoing dialogue |
| **auction** | 1→N | structured | bid → select winner → continue with winner | Competitive routing, best-fit selection |
| **discussion** | 1→N | structured | raise hand → ordered speaking | Deliberation, review, consensus |

```python
class SessionType(str, Enum):
    NOTIFICATION = "notification"
    BROADCAST = "broadcast"
    CONSULTING = "consulting"
    CONVERSATION = "conversation"
    AUCTION = "auction"
    DISCUSSION = "discussion"
```

### State Machines

Each session type defines a protocol machine with allowed states, transitions, and role-based send permissions.

**Notification:**
```
created ──send──► delivered ──ack──► completed
                            ──timeout──► expired
```
Roles: initiator (sends), recipient (acks). One message, one ack.

**Broadcast:**
```
created ──send──► delivered ──all_acked──► completed
                            ──partial_timeout──► partial
```
Roles: initiator (sends), recipients (ack individually). Hub tracks per-recipient ack status.

**Consulting:**
```
created ──request──► awaiting_response ──response──► completed
                                       ──timeout──► expired
```
Roles: initiator (sends request), recipient (sends response). Strictly one exchange. Recipient cannot proactively send. This is the direct replacement for current `hub.delegate()`.

**Conversation:**
```
created ──first_message──► active ◄──either_sends──► active
                                   ──close──► closed
                                   ──idle_timeout──► expired
```
Roles: initiator, recipient. Either side can send at any time. Most flexible 1:1 type. No turn constraints.

**Auction:**
```
created ──request──► bidding ──all_responded──► selecting ──award──► awarded ──► active (1:1 with winner) ──► completed
                              ──timeout──► selecting (with available bids)
```
Roles: initiator (sends request, selects winner), recipients (bid or pass). After selection, losers are notified and session continues as 1:1 consulting or conversation with the winner.

**Discussion:**
```
created ──topic──► gathering ──all_responded──► moderating ──set_order──► speaking ──all_spoken──► concluded
                              ──timeout──► moderating (with available hands)
```
Roles: initiator/moderator (poses topic, sets speaking order), participants (raise hand or pass, speak in turn). Moderator controls flow.

### Protocol Machine

Each session type is implemented as a protocol machine that validates interactions:

```python
@runtime_checkable
class SessionProtocol(Protocol):
    """Validates and manages session state transitions."""

    @property
    def session_type(self) -> SessionType: ...

    def validate_send(self, sender: str, envelope: Envelope, state: SessionState) -> bool:
        """Can this sender send this envelope in this state?"""
        ...

    def next_state(self, envelope: Envelope, state: SessionState) -> SessionState:
        """Compute next state after this envelope."""
        ...

    def pending_responses(self, state: SessionState) -> list[PendingResponse]:
        """What responses are expected in this state?"""
        ...

    def deadline(self, state: SessionState) -> datetime | None:
        """When does the current state expire?"""
        ...
```

The hub creates the appropriate protocol machine when a session is created. All sends go through `validate_send()` before being committed to the WAL. Invalid sends are rejected with a clear error.

### Ack and Response Detection

Two-level acknowledgment:

1. **Delivery receipt** (system-level, automatic): When the hub appends an envelope to the session WAL, it generates a `DeliveryReceipt` event. This confirms the message is durably stored and available to the recipient. No actor action needed.

2. **Processing response** (application-level, actor-driven): The recipient sends a response envelope with `reply_to` referencing the original envelope ID. The session protocol tracks pending responses and triggers `ResponseTimeout` events when deadlines are missed.

Ack semantics vary by session type:
- **notification/broadcast**: delivery receipt is sufficient
- **consulting**: processing response is required (the whole point is getting a response)
- **conversation**: no required acks (free-form)
- **auction**: bid or explicit pass required from each recipient
- **discussion**: raise_hand or explicit pass required from each recipient

### Conflict Resolution

For session types where timing matters (auction bids arriving simultaneously, discussion raise-hand conflicts): **first timestamp wins**. UUID7 provides millisecond-precision timestamps. In case of exact millisecond collision, the lower UUID7 value wins (deterministic tiebreak). This is simple, predictable, and the session protocol classes can be extended with custom resolution logic later without architectural changes.

### Public Sessions and Subscription

Multi-recipient sessions (broadcast, auction, discussion) can be configured as public:

- **Subscribe**: Receive session events (read-only). Does not make the actor a participant. Useful for monitoring, auditing.
- **Join**: Become a full participant (receive events and send). Subject to access rule validation.

Public sessions are discoverable via `peers(action="discover")` with a session filter.

### Actor Invocation Model

**Per-session push with serialized delivery.** The hub pushes messages to actors — actors do not poll. Delivery is serialized per (actor, session) pair to prevent concurrent processing of messages within the same session.

When a message arrives for Actor B in session S:

1. Hub appends the message to session WAL
2. Hub acquires a per-`(actor, session)` lock
3. If lock is free: hub invokes `actor.ask()` with the incoming message. `SessionPolicy` (or a custom assembly policy) reads session history from the WAL and assembles LLM context. The actor's LLM decides what to do — respond, create a sub-session, or do nothing.
4. If lock is held (B is already processing an earlier message in this session): message queues. When B finishes, next queued message triggers another invocation.

An actor can process messages from *different* sessions concurrently, bounded by `max_concurrent_sessions` in `PassportConstraints`.

**Context propagation:** When the hub invokes an actor within a session, it sets `_current_session_id` (a contextvar). This lets the hub auto-link sub-sessions created during the invocation, and lets assembly policies and tools know which session triggered this turn. The hub invocation is minimal — it provides the trigger, not the context:

```python
# Hub invocation (network layer) — correct
_current_session_id.set(session.id)
reply = await actor.ask(incoming_message, context=ctx)

# NOT this — network must not make assembly decisions
# history = await self.session_log(session.id).read(limit=20)
# reply = await actor.ask(incoming_message + format(history), context=ctx)
```

**Per session type:**

| Type | Invocation behavior |
|------|-------------------|
| **notification** | Hub invokes recipient for delivery. Ack is system-level (no actor response needed). |
| **broadcast** | Hub invokes each recipient for delivery. Per-recipient ack tracking. |
| **consulting** | Hub invokes recipient once with the request. Recipient responds. Session completes. |
| **conversation** | Each incoming message triggers a new `ask()`, serialized per session. Actor's LLM sees history via assembly policy and decides whether/how to respond. |
| **auction** | Hub invokes all bidders concurrently (one `ask()` each). Each responds with bid or pass. |
| **discussion** | Hub invokes all participants for raise-hand phase, then each speaker in order. |

### Participant Status

Multi-party sessions (broadcast, auction, discussion) track per-participant status:

```python
class ParticipantStatus(str, Enum):
    PENDING = "pending"           # Invited, not yet responded
    ACTIVE = "active"             # Engaged in session
    RESPONDED = "responded"       # Completed their role (bid, spoke, acked)
    PASSED = "passed"             # Explicitly declined
    FAILED = "failed"             # Invocation raised an error
    TIMED_OUT = "timed_out"       # Didn't respond within deadline
```

**Graceful degradation principle: no single participant can block a multi-party session.** Participant failure is equivalent to timeout. The protocol proceeds with available responses.

- **Auction**: `bidding → selecting` triggers when all participants reach a terminal status (responded/passed/failed/timed_out), OR when deadline expires. Failed/timed-out bidders are treated as "no bid."
- **Discussion**: If a speaker fails or times out, a `ParticipantFailed` envelope is appended to the session WAL. The moderator's LLM decides: skip to next speaker or end discussion. The protocol machine advances automatically on timeout.
- **Broadcast**: Per-recipient delivery tracking. Failed deliveries are logged but don't block other recipients.

### Sub-Session Lineage

When an actor creates a session while processing a message within another session, the new session is a **sub-session**. The hub tracks lineage automatically via `_current_session_id`.

```python
@dataclass
class Session:
    ...
    parent_session_id: UUID7 | None = None   # Immediate parent (set automatically by hub)
    root_session_id: UUID7 | None = None     # Top of the chain (computed at creation)
```

**Automatic linking:** When the hub invokes an actor within session S, it sets `_current_session_id = S.id`. When the actor creates a sub-session via the `session` tool, the hub reads `_current_session_id` and sets `parent_session_id` automatically. The actor doesn't need to pass it explicitly.

**Metadata inheritance:** Sub-sessions inherit the parent's `metadata` dict with merge semantics — child keys override parent keys on conflict.

**Root tracking:** The hub computes `root_session_id` at creation time. If `parent_session_id` is set, the hub copies the parent's `root_session_id` (or uses the parent's `id` if the parent is itself a root). This gives O(1) queries for "show me everything in this workflow":

```python
await hub.list_sessions(root=root_session_id)  # All sessions in the workflow tree
```

**Depth limiting:** `allow_sub_sessions` in `AccessRule` controls whether an actor can create sub-sessions at all. A `max_sub_session_depth` in `LimitRule` (default: 10) prevents unbounded recursion.

## Envelope

The message unit within a session. Simplified from the current Envelope — session provides most context (participants, type, trace).

```python
@dataclass
class Envelope:
    id: UUID7                            # Unique, timestamp-sortable
    session_id: UUID7                    # Which session this belongs to
    sender: str                          # Actor name

    # Content
    content: str | BaseEvent             # Message payload
    content_type: Literal["text", "event", "stream_chunk"] = "text"

    # Correlation
    reply_to: UUID7 | None = None        # Envelope this responds to
    task_id: UUID7 | None = None         # Task attribution (if part of a task)

    # Extensible
    metadata: dict[str, Any] = field(default_factory=dict)

    # Derived from UUID7
    @property
    def timestamp(self) -> datetime: ...

    # Serialization
    def to_dict(self) -> dict[str, Any]: ...

    @classmethod
    def from_dict(cls, data: dict) -> Envelope: ...
```

What's removed vs current Envelope:
- No `recipient` — session knows participants
- No `trace_id` — `session_id` serves this purpose
- No `priority` — session-level concern
- No `ttl` — session/task level concern
- No `requires_ack` — session type determines this
- No `causation_id` — `reply_to` is cleaner and more direct
- No `correlation_id` — `reply_to` chain provides correlation

### Streaming

When an actor generates a response via LLM, intermediate chunks flow through the session as envelopes with `content_type="stream_chunk"`. Subscribers receive chunks in real-time via `SessionLog.subscribe()`. The final complete response is sent as a regular `content_type="text"` envelope. Assembly policies on the receiving end decide whether to show chunks or wait for completion.

## Task

A task is a lifecycle-managed work unit within a session. Tasks solve the durability and async tracking problem — they have structured phases, TTL enforcement, progress tracking, and subscriber notification.

### Data Model

```python
class TaskStatus(str, Enum):
    CREATED = "created"
    ACTIVE = "active"
    COMPLETED = "completed"
    FAILED = "failed"
    CANCELLED = "cancelled"
    EXPIRED = "expired"
    ORPHANED = "orphaned"

@dataclass
class TaskPhase:
    name: str
    status: Literal["pending", "active", "completed", "skipped"] = "pending"
    started_at: datetime | None = None
    completed_at: datetime | None = None

@dataclass
class Task:
    id: UUID7                            # Generated by hub
    session_id: UUID7
    creator: str                         # Actor who created the task
    description: str
    status: TaskStatus = TaskStatus.CREATED
    blocking: bool = True                # Whether session waits for completion

    # Lifecycle
    ttl_seconds: int = 3600
    phases: list[TaskPhase] = field(default_factory=list)
    current_phase: int = 0               # Index into phases
    progress: float = 0.0                # 0.0 - 1.0
    result: str | None = None

    # Timestamps
    created_at: datetime = field(default_factory=datetime.now)
    updated_at: datetime = field(default_factory=datetime.now)

    @property
    def expires_at(self) -> datetime:
        return self.created_at + timedelta(seconds=self.ttl_seconds)

    @property
    def is_terminal(self) -> bool:
        return self.status in {
            TaskStatus.COMPLETED, TaskStatus.FAILED,
            TaskStatus.CANCELLED, TaskStatus.EXPIRED,
            TaskStatus.ORPHANED,
        }
```

### State Machine

```
created ──start──► active ──complete──► completed
                          ──fail──► failed
                          ──cancel──► cancelled (by initiator)
                          ──ttl_expire──► expired (by framework)
                          ──no_updates──► orphaned (by framework)
```

### Lifecycle Responsibilities

**Framework-managed (hub enforces automatically):**

| Concern | Mechanism |
|---------|-----------|
| Creation | Hub generates task_id (UUID7), validates against rules (TTL limits, concurrent task limits) |
| TTL enforcement | Background timer per task. On expiry: `TaskExpired` event, transition to `expired` |
| Phase validation | Validates phase transitions are sequential. Rejects out-of-order phase updates |
| State machine | Only allows valid transitions. Rejects invalid status changes |
| Subscriber notification | On every state change, notifies all task subscribers via session WAL |
| Deadline reminders | `TaskDeadlineApproaching` event at 80% of TTL |
| Orphan detection | If no update for configurable `orphan_timeout` (default: 5 minutes), `TaskOrphaned` event |
| Dual-write indexing | Task events written to both session WAL (chronological) and task-specific log (efficient query) |

**Actor-cooperative (actor reports via `task` tool):**

| Concern | Mechanism |
|---------|-----------|
| Progress updates | `task(action="update", task_id="...", progress=0.5)` |
| Phase advancement | `task(action="update", task_id="...", phase="next_phase")` |
| Result reporting | `task(action="update", task_id="...", status="completed", result="...")` |
| Failure reporting | `task(action="update", task_id="...", status="failed", result="Error: ...")` |
| Streaming output | Send envelopes to session with `task_id` attribution |

### Blocking vs Background

- **Blocking**: Session protocol holds the initiator in a "waiting" state. The session tool call blocks until the task completes (or times out). The initiator cannot send new messages in this session until the task resolves. Use case: "process this and give me the result."
- **Background**: Session stays active. The initiator can send new messages, start new tasks, or do other work. Task events stream in asynchronously via subscription. Use case: "start analyzing this, I'll check back later."

### Checkpoint and Recovery

Phases serve as checkpoints. If an actor crashes mid-task:
1. Hub detects orphan (no updates for `orphan_timeout`)
2. Hub fires `TaskOrphaned` event to session WAL
3. A recovery actor (or the same actor restarted) reads the task log to find last completed phase
4. Resumes from the last checkpoint

The task log contains the full history of phase transitions and progress updates, making recovery deterministic.

## SessionLog

The WAL protocol for append-only session event storage. Separate from `KnowledgeStore` because the access patterns are fundamentally different: KnowledgeStore is read/write/overwrite; SessionLog is append/sequential-read/subscribe.

```python
@runtime_checkable
class SessionLog(Protocol):
    """Append-only sequential log for session events."""

    async def append(self, entry: Envelope) -> None:
        """Append envelope to log."""
        ...

    async def read(
        self,
        *,
        after: UUID7 | None = None,
        limit: int = 100,
    ) -> list[Envelope]:
        """Read entries after the given ID (or from start). Chronological order."""
        ...

    async def subscribe(
        self,
        callback: Callable[[Envelope], Awaitable[None]],
        *,
        after: UUID7 | None = None,
    ) -> str:
        """Subscribe to new entries. Optionally replay from a point. Returns sub_id."""
        ...

    async def unsubscribe(self, sub_id: str) -> None: ...

    @property
    async def last_id(self) -> UUID7 | None:
        """ID of the most recent entry."""
        ...
```

Using UUID7 as the cursor means the log is naturally ordered, globally unique, and doesn't need a centralized counter. `read(after=last_seen_id)` is the polling API. `subscribe()` is the push API.

### Implementations

| Backend | Class | Use Case |
|---------|-------|----------|
| In-memory | `MemorySessionLog` | Development, testing |
| Filesystem | `FileSessionLog` | Single-machine persistent (append-only file) |
| Redis Streams | `RedisSessionLog` | Distributed, production |

### Dual-Write for Tasks

When a task event is appended to the session log, the hub also appends it to the task-specific log (a separate `SessionLog` instance keyed by `session_id/tasks/task_id`). The session log has the complete chronological record. The task log is a materialized index for efficient task queries. This is a write-time optimization:

```python
# In hub's send logic:
await session_log.append(envelope)
if envelope.task_id:
    await task_log(envelope.task_id).append(envelope)
```

## Persistence Architecture

`HubStore` composes `KnowledgeStore` (static data) + `SessionLog` factory (dynamic data):

```python
class HubStore:
    """Unified storage for hub state."""

    def __init__(
        self,
        store: KnowledgeStore,
        log_factory: Callable[[str], SessionLog],
    ):
        self._store = store
        self._log_factory = log_factory
        self._logs: dict[str, SessionLog] = {}

    # Static data (identity, rules, config, session metadata)
    async def read(self, path: str) -> str | None: ...
    async def write(self, path: str, content: str) -> None: ...
    async def list(self, path: str) -> list[str]: ...
    async def delete(self, path: str) -> None: ...
    async def exists(self, path: str) -> bool: ...

    # Session logs
    def session_log(self, session_id: str) -> SessionLog:
        """Get or create the WAL for a session."""
        if session_id not in self._logs:
            self._logs[session_id] = self._log_factory(session_id)
        return self._logs[session_id]

    def task_log(self, session_id: str, task_id: str) -> SessionLog:
        """Get or create the task-specific index log."""
        key = f"{session_id}/tasks/{task_id}"
        if key not in self._logs:
            self._logs[key] = self._log_factory(key)
        return self._logs[key]
```

### Backend Combinations

| Environment | KnowledgeStore | SessionLog |
|-------------|---------------|------------|
| Development | `MemoryKnowledgeStore` | `MemorySessionLog` |
| Single-machine | `DiskKnowledgeStore` | `FileSessionLog` |
| Production | `S3KnowledgeStore` or `RedisKnowledgeStore` | `RedisSessionLog` |

The per-actor `KnowledgeStore` is unchanged — it remains the actor's private filesystem for its own knowledge, separate from the hub's state.

## Gateway Protocol (Remote Actors)

When actors are on different servers, hubs communicate via the Gateway Protocol — a WebSocket-based protocol for cross-hub session management, with HTTP for stateless discovery.

### Design Principles

1. **Initiator-owned sessions.** The hub that creates a session owns all session state (WAL, metadata, task tracking). The remote hub is stateless for the session — it delivers messages to its local actor and forwards responses. This simplifies recovery: if the remote hub restarts, the session resumes because the initiating hub has all state.

2. **Transform-at-boundary.** Outbound transforms run on the sender's hub. Inbound transforms run on the recipient's hub. The wire carries the envelope between the two transform stages.

3. **WebSocket for all session traffic.** One persistent bidirectional WebSocket per hub-to-hub link, multiplexed across sessions via `session_id` in each message frame. HTTP is only used for stateless operations (manifest discovery, identity inspection) where a persistent connection isn't warranted.

Why WebSocket over HTTP+SSE:
- **Bidirectional on one connection.** SSE is read-only, requiring a separate POST channel for sends — two connections doing what one WebSocket does.
- **Lower overhead.** No repeated HTTP headers for every envelope exchange. Conversation/discussion sessions can involve hundreds of messages.
- **Simpler reconnection.** One connection to manage, not two. Reconnect with `last_envelope_id` to resume.
- **Natural fit for streaming.** LLM response chunks, task progress, and session events all flow on the same channel.

### Connection Establishment

Hub A discovers Hub B at a known endpoint and retrieves the manifest:

```
GET /api/v1/gateway/manifest
```

Response:
```json
{
  "hub_id": "uuid7",
  "actors": [
    {
      "name": "acme:writer:v1",
      "capabilities": ["writing", "editing"],
      "tags": ["writing"],
      "session_types": ["consulting", "conversation"]
    }
  ],
  "version": "1.0"
}
```

The manifest is the remote hub's public interface. Only actors whose rules expose identity are listed. Hub A creates `RemoteActor` proxies for discovered actors and registers them locally. Remote actors appear identical to local actors from the perspective of other local actors.

After discovery, Hub A opens a WebSocket connection for all session traffic:

```
WS /api/v1/gateway/ws?hub_id={hub_a_id}&last_envelope_id={id}
```

The `last_envelope_id` parameter enables resumption after disconnects — the remote hub replays any envelopes sent after that ID.

### Endpoints

```
# HTTP (stateless discovery — no persistent connection needed)
GET   /api/v1/gateway/manifest                              # Hub manifest
GET   /api/v1/gateway/actors/{name}/identity                 # Actor public identity
GET   /api/v1/gateway/actors/{name}/identity/{section}       # passport, resume, or skill

# WebSocket (all session communication — single multiplexed connection)
WS    /api/v1/gateway/ws                                     # Bidirectional session traffic
```

### WebSocket Message Format

All session communication flows as JSON messages over the WebSocket, multiplexed by `session_id`:

```json
{
  "type": "envelope",
  "session_id": "uuid7",
  "envelope": { ... }
}
```

Message types:

| Type | Direction | Purpose |
|------|-----------|---------|
| `session_create` | A→B | Create inbound session |
| `session_create_ack` | B→A | Confirm session creation (or reject with error) |
| `session_close` | either | Close session |
| `envelope` | either | Send/receive envelope (messages, chunks, acks) |
| `task_create` | either | Create task within session |
| `task_update` | either | Update task status/progress |
| `task_status` | B→A | Task status response |
| `error` | either | Error for a specific session |
| `ping` / `pong` | either | Keepalive |

Each message carries `session_id` for multiplexing. One WebSocket connection handles all sessions between two hubs simultaneously.

### Session Flow (Cross-Hub Consulting)

```
Actor A (Hub 1)                  Hub 1                    WS                    Hub 2                  Actor B (Hub 2)
────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
session(create, consulting, [B])
      │
      ├──► Validates A's rules
      ├──► Creates session locally
      ├──► A's outbound transforms
      ├────────────────────────── session_create ──────────────────────►
      │                                                                 Validates B's rules
      │                                                                 B's inbound transforms
      │                                                                 Delivers to B
      │                                                                           │
      │                                                                 B processes, responds
      │                                                                 B's outbound transforms
      ◄────────────────────────── envelope (response) ────────────────
      ├──► A's inbound transforms
      ├──► Appended to session WAL
      └──► Result returned to Actor A
```

### Session Flow (Cross-Hub Conversation)

```
Actor A (Hub 1)                  Hub 1                    WS                    Hub 2                  Actor B (Hub 2)
────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
session(create, conversation, [B])
      │
      ├──► Creates session
      ├────────────────────────── session_create ──────────────────────►
      ◄────────────────────────── session_create_ack ─────────────────
      │
session(send, "Hello")
      ├────────────────────────── envelope ───────────────────────────►
      │                                                                 Delivers to B
      │                                                                           │
      │                                                                 B responds
      ◄────────────────────────── envelope ───────────────────────────
      ├──► Appended to WAL
      │
      │    (either side can send at any time — bidirectional on same WS)
      │
      ◄────────────────────────── envelope (B proactively sends) ─────
      ├────────────────────────── envelope ───────────────────────────►
      ...
```

### Reconnection

If the WebSocket disconnects:

1. Hub A reconnects: `WS /api/v1/gateway/ws?hub_id={id}&last_envelope_id={last_received}`
2. Hub B replays any envelopes sent after `last_envelope_id`
3. Active sessions resume automatically — no re-creation needed
4. The initiating hub has all session state, so nothing is lost

### Authentication

Initially: shared secret (API key) passed as query parameter on WebSocket upgrade or in the HTTP `Authorization` header for discovery endpoints. The hub config stores trusted hub keys.

Future: mTLS for hub-to-hub, OAuth2 for user-initiated connections, or hub-level certificates for zero-trust networks.

### Error Handling

| Scenario | Behavior |
|----------|----------|
| Remote hub unreachable | `DeliveryFailure` event on hub stream. Retry WebSocket connection with backoff. |
| WebSocket disconnects mid-session | Auto-reconnect with `last_envelope_id`. Session state preserved (initiator-owned). Pending envelopes replayed. |
| Response timeout | `ResponseTimeout` event in session WAL. Actor notified via assembly policy. |
| Remote actor not found | `session_create_ack` with error. Session creation rejected. |
| Rule violation on remote hub | `session_create_ack` or `error` message with reason. `RuleViolation` event on hub stream. |

### Federation and Self-Loop Prevention

Each hub has a unique `hub_id` (UUID7). When connecting hubs, the manifest exchange includes hub_id. If Hub A connects to Hub B and discovers an actor that is a `RemoteActor` proxy pointing back to Hub A (or to a hub that Hub A already connects to), it skips that actor. This prevents self-loops.

The current `/discover` endpoint already filters out remote proxies — only local actors are exposed. This behavior carries forward into the gateway manifest.

For the case where a human registers the same actor on two different hubs that connect to each other: actor names include `owner:name:version`. If the same owner registers the same name on both hubs, the manifest will show duplicates. Hub A can detect this (same name, different hub_id) and warn the user. Framework does not auto-resolve — this is a configuration error that the user should fix.

## System Extensibility: Operators

**Name: Operator** (inspired by Kubernetes Operators — observe state, take reconciliation action).

Operators replace the old Plugin concept for system-level concerns. They are NOT routing interceptors (that's now handled by rules/transforms). They are autonomous background processes that observe hub events and take action.

**Alternative name considered:** Sentinel. "Operator" was chosen because it's an established pattern with clear semantics: observe desired vs actual state, reconcile.

### Concept (Design Only — Not Implemented Now)

An Operator:
1. Subscribes to `hub.stream` for relevant events
2. Has access to hub API (create sessions, register/unregister actors, update rules, query sessions/tasks)
3. Runs as a background async task
4. Has its own state (via hub store or its own KnowledgeStore)

```python
# Conceptual — not implemented in this phase
class Operator(Protocol):
    name: str

    async def start(self, hub: Hub) -> None:
        """Begin operating. Subscribe to events, start background tasks."""
        ...

    async def stop(self) -> None:
        """Stop operating. Clean up."""
        ...
```

### Example Operators (Future)

| Operator | Observes | Acts |
|----------|----------|------|
| `AutoScaleOperator` | Session/task creation rate, queue depth | Registers actor replicas, load-balances via rule updates |
| `CircuitBreakerOperator` | Task failures, response timeouts | Suspends failing actors (update rule), auto-recovers after cooldown |
| `AuditOperator` | All hub events | Writes audit log, generates compliance reports |
| `CostTrackingOperator` | Task completions with usage metadata | Tracks per-actor cost, enforces budget limits via rule updates |
| `HealthCheckOperator` | Periodic timer | Pings actors via notification sessions, marks unhealthy actors |

Operators are a natural AG2 Cloud feature — they provide managed infrastructure intelligence on top of the open-source hub. The hub stream + hub API provide everything an Operator needs. No special protocol beyond "subscribe and call API."

## Actor Protocol & Notification Mechanism

### Problem

The current implementation has the Hub directly invoke `actor.ask()` when messages arrive. This conflates two responsibilities:

1. **Routing**: Delivering messages to the right inbox
2. **Execution**: Invoking the actor's LLM

This breaks when:
- Actor is busy (concurrent session processing)
- Actor is remote (can't just call `.ask()`)
- Actor needs custom handling per session type
- We want to track delivery status separate from processing status

### Design Principles

1. **Hub is mail router, not actor manager.** The hub delivers messages to inboxes and calls `notify()`. It never invokes actor LLM calls.
2. **Actor decides how to handle notifications.** Default implementations per session type, but fully overridable.
3. **Three-state response model.** `ACK`, `ACK_AND_RESPONDING`, `REJECT` — sender knows whether to wait.
4. **Durable inbox.** Messages persist; crashes don't lose notifications.

### NotifyResponse

```python
class NotifyResponse(str, Enum):
    ACK = "ack"                         # Received, no response coming
    ACK_AND_RESPONDING = "ack_responding"  # Received, response will be sent
    REJECT = "reject"                   # Rejected (busy, unauthorized, etc.)
```

**Why three states?**

- `ACK`: For notification/broadcast — actor acknowledges receipt, done
- `ACK_AND_RESPONDING`: For consulting/conversation — actor will generate a response. Sender can decide to block-wait or continue.
- `REJECT`: Actor refuses (at capacity, wrong session type, etc.)

### Actor Protocol

```python
@runtime_checkable
class Actor(Protocol):
    """Protocol for network-participating actors."""
    
    @property
    def identity(self) -> Identity: ...
    
    async def notify(
        self,
        entry: InboxEntry,
        hub: "Hub",
    ) -> NotifyResponse:
        """Called when a message arrives for this actor.
        
        The actor can:
        - Return ACK and do nothing (fire-and-forget received)
        - Return ACK_AND_RESPONDING and spawn async work to respond
        - Return REJECT if it cannot handle this
        
        The hub NEVER calls the actor's LLM. The actor decides.
        """
        ...
```

### Inbox Storage

```

├── actors/
│   └── {actor_id}/

│           ├── processing.jsonl    # Currently being handled
```

Benefits:
- **Durability**: Crashes don't lose messages
- **Observability**: Query an actor's backlog
- **Rate control**: Actor sees queue depth, can REJECT if overloaded
- **Recovery**: On restart, replay `pending` + `processing` entries

### AgentActor — Default Implementation

```python
class AgentActor:
    """Wraps an Agent to implement the Actor protocol with sensible defaults."""
    
    def __init__(
        self,
        agent: Agent,
        identity: Identity,
        *,
        notify_handler: Callable[..., Awaitable[NotifyResponse]] | None = None,
    ):
        self.agent = agent
        self._identity = identity
        self._notify_handler = notify_handler or self._default_notify
    
    @property
    def identity(self) -> Identity:
        return self._identity
    
    async def notify(self, entry: InboxEntry, hub: Hub) -> NotifyResponse:
        return await self._notify_handler(entry, hub, self.agent)
    
    async def _default_notify(
        self, entry: InboxEntry, hub: Hub, agent: Agent
    ) -> NotifyResponse:
        """Default behavior based on session type."""
        session = await hub.get_session(entry.session_id)
        
        if session.type == SessionType.NOTIFICATION:
            return NotifyResponse.ACK
        
        if session.type == SessionType.BROADCAST:
            return NotifyResponse.ACK
        
        if session.type in (SessionType.CONSULTING, SessionType.CONVERSATION):
            asyncio.create_task(self._respond_async(entry, hub, agent))
            return NotifyResponse.ACK_AND_RESPONDING
        
        if session.type == SessionType.AUCTION:
            asyncio.create_task(self._bid_async(entry, hub, agent))
            return NotifyResponse.ACK_AND_RESPONDING
        
        if session.type == SessionType.DISCUSSION:
            asyncio.create_task(self._discuss_async(entry, hub, agent))
            return NotifyResponse.ACK_AND_RESPONDING
        
        return NotifyResponse.REJECT
    
    async def _respond_async(
        self, entry: InboxEntry, hub: Hub, agent: Agent
    ) -> None:
        """Background task: invoke agent and send response."""
        try:
            reply = await agent.ask(
                entry.envelope.content,
                dependencies={"hub": hub},
            )
            await hub.send(
                entry.session_id,
                self._identity.passport.name,
                str(reply.response.content),
                reply_to=entry.envelope.id,
            )
        except Exception as e:
            await hub._emit(ParticipantFailed(
                session_id=str(entry.session_id),
                participant=self._identity.passport.name,
                reason=str(e),
            ))
```

### Hub Changes

The hub's `_invoke_actor()` is replaced by `_deliver_to_inbox()`:

```python
async def _deliver_to_inbox(
    self,
    actor_name: str,
    envelope: Envelope,
) -> NotifyResponse:
    """Deliver envelope to actor's inbox and call notify()."""
    entry = self._actors.get(actor_name)
    if not entry:
        return NotifyResponse.REJECT
    
    session = await self.get_session(envelope.session_id)
    
    # Write to inbox (durability)
    inbox = self._store.actor_inbox(actor_name)
    inbox_entry = InboxEntry(
        id=uuid7(),
        session_id=envelope.session_id,
        session_type=session.type,
        envelope=envelope,
        received_at=datetime.now(tz=timezone.utc),
        status="pending",
    )
    await inbox.append(inbox_entry)
    
    # Acquire per-(actor, session) lock for serialized delivery
    async with self._get_mailbox_lock(actor_name, envelope.session_id):
        inbox_entry.status = "processing"
        
        # Call notify — actor decides what to do
        response = await entry.actor.notify(inbox_entry, self)
        
        inbox_entry.notify_response = response
        inbox_entry.status = "processed"
    
    return response
```

## Remote Actor Registration Protocol

### Problem

Current design only supports pull-based discovery:
- Hub A connects to Hub B
- Hub A discovers Hub B's actors via manifest
- Hub A creates local proxies

This doesn't support:
- Actor on Server X registering with Hub on Server Y (push model)
- True transparency — local code shouldn't know if actor is local/remote
- Scaling to many actors across many servers without full mesh

### Design Principle

**"Your device can register to any provider signal tower."** An actor should be able to register itself to any hub, regardless of where it runs.

### Registration Protocol

#### Push Registration Endpoint

```
    "hub_id": "uuid7",
```

#### Unified ActorEntry

```python
@dataclass
class ActorEntry:
    """Unified entry for local and remote actors."""
    
    identity: Identity
    rule: Rule
    transforms: dict[str, Callable[..., Awaitable[Envelope]]]
    
    # Location (exactly one is set)
    local_actor: Actor | None = None       # Local: implements Actor protocol
    remote_endpoint: RemoteEndpoint | None = None  # Remote: connection info
    
    @property
    def is_local(self) -> bool:
        return self.local_actor is not None
    
    async def notify(self, entry: InboxEntry, hub: Hub) -> NotifyResponse:
        """Unified notify — works for both local and remote."""
        if self.local_actor:
            return await self.local_actor.notify(entry, hub)
        else:
            return await self._notify_remote(entry, hub)
    
    async def _notify_remote(
        self, entry: InboxEntry, hub: Hub
    ) -> NotifyResponse:
        """Forward notify to remote actor via WebSocket."""
        endpoint = self.remote_endpoint
        
        try:
            await endpoint.send({
                "type": "notify",
                "inbox_entry": entry.to_dict(),
            })
            
            response = await endpoint.receive_notify_response(
                entry.id, timeout=30.0
            )
            return NotifyResponse(response["response"])
        
        except (asyncio.TimeoutError, ConnectionError):
            endpoint.mark_disconnected()
            return NotifyResponse.REJECT
```

### ActorServer (for hosting remote actors)

```python
class ActorServer:
    """Hosts actors and connects them to remote hubs."""
    
    def __init__(self, host: str = "0.0.0.0", port: int = 8901):
        self._actors: dict[str, Actor] = {}
        self._hub_connections: dict[str, WebSocket] = {}
    
    def add_actor(self, actor: Actor) -> None:
        """Add an actor to be hosted by this server."""
        self._actors[actor.identity.passport.name] = actor
    
    async def register_with_hub(
        self,
        actor: Actor,
        hub_endpoint: str,
        *,
        api_key: str = "",
    ) -> None:
        """Register a local actor with a remote hub."""
        async with aiohttp.ClientSession() as session:
            resp = await session.post(
                f"{hub_endpoint}/api/v1/gateway/actors/register",
                json={
                    "identity": actor.identity.to_dict(),
                    "rule": Rule().to_dict(),
                    "notify_endpoint": f"wss://{self.host}:{self.port}/notify",
                },
                headers={"Authorization": f"Bearer {api_key}"}
            )
            data = await resp.json()
            
            if data["status"] != "registered":
                raise RegistrationError(data.get("error"))
            
            # Connect WebSocket for receiving notifies
            ws = await session.ws_connect(
                data["session_endpoint"],
                headers={"Authorization": f"Bearer {api_key}"}
            )
            self._hub_connections[data["hub_id"]] = ws
            
            asyncio.create_task(self._listen_for_notifies(data["hub_id"]))
    
    async def _listen_for_notifies(self, hub_id: str) -> None:
        """Handle incoming notify messages from a hub."""
        ws = self._hub_connections[hub_id]
        
        async for msg in ws:
            if msg.type == aiohttp.WSMsgType.TEXT:
                data = json.loads(msg.data)
                
                if data["type"] == "notify":
                    actor = self._actors.get(data["actor_id"])
                    if actor:
                        entry = InboxEntry.from_dict(data["inbox_entry"])
                        hub_proxy = RemoteHubProxy(ws, hub_id)
                        response = await actor.notify(entry, hub_proxy)
                        
                        await ws.send_json({
                            "type": "notify_response",
                            "inbox_entry_id": str(entry.id),
                            "response": response.value,
                        })
```
